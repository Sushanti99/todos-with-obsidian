# Brain — Personal Knowledge Agent

> A personal knowledge management system where an Obsidian vault is the database, an agentic CLI (Claude Code / ChatGPT Codex / Gemini CLI) is the brain, and a lightweight web chat interface ties it all together.

---

## Overview

Brain is a three-layer personal operating system:

1. **Data Layer** — An Obsidian vault of markdown files. This is the single source of truth for everything: daily todos, life interests, references, and conversation history.
2. **Agent Layer** — An agentic CLI tool (Claude Code, ChatGPT Codex, or Gemini CLI) that reads, writes, searches, and reasons over the vault. Configured at startup.
3. **Interface Layer** — A bare-minimal web chat UI served locally. The user talks to the agent through this. The agent operates on the vault. Changes sync to all devices via Obsidian's native sync.

### Design Principles

- **Obsidian is the database.** No other storage. No SQLite. No Redis. Just markdown files in folders.
- **The agent is swappable.** The system supports Claude Code, ChatGPT Codex, and Gemini CLI as interchangeable agent backends, selected at startup via config.
- **The interface is disposable.** A single HTML file with vanilla JS. No React, no build step, no npm for the frontend. It must load instantly.
- **V1 runs locally.** Your laptop is the server. The vault lives on your filesystem. The agent runs in your terminal.
- **V2 deploys to a VPS.** Same codebase, deployed to Hetzner/Contabo/Fly.io with obsidian-headless for vault sync. Packaged as a real open-source project.

## V1 Scope — Local Setup

### What V1 Does

- User runs `brain start` from the terminal.
- A web chat UI opens at `http://localhost:3000`.
- User types messages. The server spawns the configured agent CLI (`claude -p`, `codex`, or `gemini`) pointed at the Obsidian vault.
- The agent can read any file in the vault, write/edit files, search across the vault, and run shell commands scoped to the vault directory.
- Responses stream back to the chat UI in real-time.
- Conversation sessions are auto-saved as summarized logs in `thoughts/`.

### What V1 Does NOT Do

- No mobile access (unless user sets up Tailscale themselves).
- No multi-user support.
- No Obsidian plugin integration.
- No voice input.
- No background/scheduled tasks (e.g., no "remind me at 3pm").

## Vault Structure

```
~/vault/
├── daily/
│   ├── 2026-04-11.md          # Today's note
│   ├── 2026-04-10.md          # Yesterday (archived)
│   └── ...
├── core/
│   ├── <user-defined>.md      # One file per life branch
│   ├── <user-defined>.md      # e.g., health, finance, project-x
│   └── ...                    # User creates ~10 of these
├── references/
│   ├── <auto-generated>.md    # Ingested articles, tweets, transcripts
│   └── ...
├── thoughts/
│   ├── 2026-04-11-session-1.md  # Auto-saved conversation summary
│   └── ...
├── system/
│   ├── CLAUDE.md              # System prompt for the agent
│   └── brain.config.yaml      # Brain configuration
└── .obsidian/                  # Obsidian settings (ignored by agent)
```

### Folder Contracts

#### `daily/`

- **Format:** One file per day, named `YYYY-MM-DD.md`.
- **Content:** A todo list for the day. Tasks use standard markdown checkboxes (`- [ ]` / `- [x]`).
- **Lifecycle:** The agent creates today's file if it doesn't exist. Completed days are left as-is (archived by date). The agent can review past days for patterns.
- **Example:**

```markdown
# 2026-04-11

## To Do
- [ ] Review PR for wallet service
- [ ] Call dentist
- [x] Morning workout
- [ ] Read chapter 5 of current book

## Notes
- Pushed back deploy to Monday, infra issues
```

#### `core/`

- **Format:** One markdown file per "branch of life" or area of active interest.
- **Content:** Free-form, maintained over time. Can contain goals, notes, ongoing thoughts, links, structured sections — whatever the user builds up.
- **Lifecycle:** The user creates these files initially (during setup or organically). The agent can append to them, reorganize sections, and cross-reference between them — but should not delete content without explicit instruction.
- **Naming:** User-defined. No prescribed names. Examples: `health.md`, `fere-ai.md`, `reading.md`, `investments.md`.

#### `references/`

- **Format:** One file per reference item. Auto-named by the agent based on content.
- **Content:** Raw or lightly processed external content — articles, tweets, video transcripts, quotes, research snippets. Each file should have frontmatter with metadata.
- **Ingestion flow:** User pastes content into the chat → agent creates a new file in `references/` → agent analyzes the content and adds frontmatter (source, date, tags, related core notes) → agent suggests which `core/` files this relates to and optionally appends a backlink or summary to the relevant core note.
- **Frontmatter template:**

```markdown
---
source: "https://example.com/article"
type: article | tweet | transcript | snippet | paper
date_added: 2026-04-11
tags: [ai, crypto, defi]
related_core: [fere-ai, investments]
---

# Title or First Line

Content goes here...
```

- **Best practices for the agent (include in system prompt):**
  - When a user pastes raw text without context, ask what it's about before filing.
  - When a URL is provided, extract the key content and summarize — don't just dump the raw HTML.
  - Always suggest which `core/` notes the reference maps to. The user can confirm or override.
  - Use descriptive filenames: `2026-04-11-defi-yield-strategies.md`, not `ref-001.md`.
  - When a reference clearly enriches an existing core note, offer to append a summary + link to that core note.
  - Periodically (when asked), review `references/` for orphaned items that aren't linked to any core note.

#### `thoughts/`

- **Format:** One file per conversation session, auto-generated.
- **Naming:** `YYYY-MM-DD-session-N.md` where N increments per day.
- **Content:** A summarized log of the conversation, not a raw transcript. The agent generates this at the end of each session (or when the user closes the chat).
- **Summary format:**

```markdown
---
date: 2026-04-11
session: 1
agent: claude-code
duration: ~15 min
---

# Session Summary

## Topics Discussed
- Reviewed daily todos, marked 3 complete
- Discussed investment thesis on L2 scaling
- Ingested article on Eigenlayer restaking → filed in references/

## Decisions Made
- Decided to pause DeFi yield research until after launch
- Added "review Eigenlayer docs" to tomorrow's daily

## Files Modified
- daily/2026-04-11.md — marked tasks complete
- core/investments.md — added Eigenlayer section
- references/2026-04-11-eigenlayer-restaking.md — new file
```

#### `system/`

- **`CLAUDE.md`** — The system prompt / personality file for the agent. This is read by Claude Code automatically when it `cd`s into the vault. For Codex and Gemini, the server injects this as a preamble.
- **`brain.config.yaml`** — Configuration for the Brain server.

## Server Implementation

### Tech Stack

- **Runtime:** Node.js 22+
- **HTTP:** Express.js (minimal, just serves static files + WebSocket upgrade)
- **WebSocket:** `ws` library (streaming agent responses)
- **Process management:** Node `child_process.spawn` (spawning agent CLI)
- **No database.** Session state lives in memory during the session, persisted as markdown in `thoughts/`.

### File Structure

```

├── server/
│   ├── index.js              # Entry point — Express + WebSocket server
│   ├── agent.js              # Agent spawner — handles claude/codex/gemini
│   ├── session.js            # Session manager — tracks conversation history
│   ├── config.js             # Reads brain.config.yaml
│   ├── prompt-builder.js     # Builds context-aware prompts for the agent
│   └── summarizer.js         # Generates session summaries for thoughts/
├── web/
│   ├── index.html            # Single-file chat UI (HTML + CSS + JS inline)
│   └── favicon.ico           # Optional
├── bin/

├── package.json
├── README.md
└── LICENSE
```

### Core Modules

#### `agent.js` — Agent Spawner

Responsibilities:
- Reads `brain.config.yaml` to determine which agent to use.
- Spawns the agent CLI as a child process, pointed at the vault directory.
- For Claude Code: runs `claude -p "<prompt>" --allowedTools "Read,Edit,Bash,Glob,Grep" --output-format stream-json` in the vault directory.
- For Codex: runs `codex --quiet "<prompt>"` in the vault directory.
- For Gemini CLI: runs `gemini "<prompt>"` in the vault directory.
- Streams stdout back to the caller (WebSocket handler) chunk by chunk.
- Handles process exit, errors, and timeouts.

Key behaviors:
- **Prompt injection:** Before sending the user's message to the agent, prepend context from the vault:
  - The contents of `system/CLAUDE.md` (or equivalent system prompt).
  - Today's daily note (`daily/YYYY-MM-DD.md`) for task awareness.
  - A listing of `core/` filenames so the agent knows what branches exist.
  - Recent conversation history from the current session (last N turns).
- **Working directory:** Always `cwd: vault_path` so the agent can read/write vault files directly.
- **Timeout:** Kill the agent process after 120 seconds if no output is received.

#### `session.js` — Session Manager

Responsibilities:
- Maintains an in-memory conversation history for the current session: `[{role: "user", content: "..."}, {role: "assistant", content: "..."}, ...]`.
- Assigns a session ID on first message of the day: `YYYY-MM-DD-session-N`.
- On session close (WebSocket disconnect or explicit `/end` command):
  - Calls the agent one final time with a summarization prompt.
  - Writes the summary to `thoughts/YYYY-MM-DD-session-N.md`.
- Provides `getRecentHistory(n)` to the prompt builder for context injection.

#### `prompt-builder.js` — Context-Aware Prompt Construction

Responsibilities:
- Reads vault state to build rich context for the agent.
- For each user message, constructs a prompt like:

```
[System prompt from CLAUDE.md]

[VAULT CONTEXT]
Today's date: 2026-04-11

Today's daily note:
---
{contents of daily/2026-04-11.md, or "No daily note yet."}
---

Core notes available: health.md, fere-ai.md, investments.md, reading.md, ...

Recent conversation:
User: {previous message}
Assistant: {previous response}
...

[USER MESSAGE]
{current user message}
```

- For Claude Code specifically: Claude Code reads `CLAUDE.md` automatically from the working directory, so the system prompt doesn't need to be injected manually. But the vault context (today's daily, core listing, conversation history) is still prepended to the user's message.
- For Codex and Gemini: The full system prompt + vault context is prepended since these tools don't have an equivalent of `CLAUDE.md` auto-loading.

#### `summarizer.js` — Session Summary Generator

Responsibilities:
- Called when a session ends.
- Takes the full conversation history and sends it to the agent with a summarization prompt:

```
Summarize this conversation session for archival. Include:
1. Topics discussed (bullet points)
2. Decisions made
3. Files modified (list paths)
4. Any action items that came up

Format as markdown with frontmatter. Keep it concise — this is a memory log, not a transcript.
```

- Writes the output to `thoughts/YYYY-MM-DD-session-N.md`.
- If the summary generation fails (agent timeout, etc.), falls back to saving a raw message log.

## Chat UI

### Requirements

- **Single HTML file.** All CSS and JS inline. No build step. No external dependencies except a markdown rendering library loaded from CDN.
- **Dark theme.** Dark background, light text. Minimal. Matches the "AI x crypto" aesthetic.
- **Markdown rendering.** Agent responses rendered as markdown (code blocks, headers, lists, bold/italic).
- **Streaming.** Responses appear token-by-token as the agent streams them.
- **Input.** Text input at the bottom. Send on Enter. Shift+Enter for newline. Support pasting long text (for dumping articles/tweets).
- **Session controls.** A small header with: session ID, agent name, a "New Session" button, and a "End & Summarize" button.
- **No auth.** V1 runs locally — no login needed.

### Technical Approach

- WebSocket connection to `ws://localhost:3000/ws`.
- Messages sent as JSON: `{ "type": "message", "content": "user's text" }`.
- Server streams back: `{ "type": "chunk", "content": "partial response" }` and `{ "type": "done", "content": "full response" }`.
- Markdown rendered client-side using `marked.js` (loaded from CDN).
- Code blocks syntax-highlighted using `highlight.js` (loaded from CDN).
- Chat history stored in-memory on the client (lost on page refresh — the server has the canonical history).

## System Prompt (`CLAUDE.md`)

This file lives at `~/vault/system/CLAUDE.md` and defines the agent's personality, rules, and operating procedures. It is the most important file in the system.

### Contents to Include

```markdown
# Brain — System Prompt

You are Brain, a personal knowledge agent. You operate on an Obsidian vault
that contains the user's daily notes, core life interests, references, and
thought logs.

## Your Role

- You are a personal assistant, thought partner, and knowledge organizer.
- You have full read/write access to the vault. Use it.
- You are not a generic chatbot. You know the user's interests, goals, and
  context because it's all in the vault.

## Vault Structure

- `daily/` — Daily todo notes. Format: YYYY-MM-DD.md with checkbox tasks.
- `core/` — Long-running notes on life branches. One file per interest area.
- `references/` — Ingested external content (articles, tweets, snippets).
- `thoughts/` — Conversation logs. Don't edit these.
- `system/` — This file and config. Don't edit these.

## Operating Rules

### Daily Notes
- When asked "what's on my plate" or similar, read today's daily note.
- If today's daily note doesn't exist, create it.
- When tasks are completed, mark them with `[x]`.
- At the end of a session, offer to create tomorrow's daily note with
  carryover tasks.

### Core Notes
- Before answering questions about a topic, check if a relevant core note
  exists and read it for context.
- When new information comes up about a core topic, offer to append it to
  the relevant core note.
- Never delete content from core notes without explicit instruction.
- When cross-references exist between core notes, use wiki-links: [[filename]].

### References
- When the user pastes raw content (article, tweet, etc.):
  1. Ask what it's about if not obvious from context.
  2. Create a new file in references/ with descriptive name and frontmatter.
  3. Identify which core/ notes it relates to.
  4. Offer to append a summary + backlink to the relevant core note(s).
- Use frontmatter: source, type, date_added, tags, related_core.
- Naming: YYYY-MM-DD-descriptive-slug.md

### Conversations & Memory
- Reference past sessions in thoughts/ when relevant.
- If the user asks "what did we discuss about X", search thoughts/ and core/.
- Build on previous conversations rather than starting from scratch.

### Tone & Style
- Direct, concise, no fluff.
- Use the user's terminology and context from their notes.
- When suggesting edits to files, show what you're changing.
- Don't over-explain things the user already knows (check their core notes
  for expertise level).
```

## CLI Interface

### Commands

```bash
# Initialize a new vault with the Brain structure
brain init [--path ~/vault]
# Creates: daily/, core/, references/, thoughts/, system/
# Creates: system/CLAUDE.md (default system prompt)
# Creates: system/brain.config.yaml (default config)

# Start the Brain server
brain start [--port 3000] [--agent claude-code]
# Reads config from system/brain.config.yaml
# Starts Express + WebSocket server
# Opens browser to http://localhost:3000

# Check configuration and agent availability
brain status
# Shows: vault path, configured agent, agent CLI version, port
```

### `brain init` Flow

1. Ask for vault path (default: `~/vault`). If the directory exists and has markdown files, confirm before restructuring.
2. Create folder structure: `daily/`, `core/`, `references/`, `thoughts/`, `system/`.
3. Write default `system/CLAUDE.md`.
4. Write default `system/brain.config.yaml`.
5. Create today's daily note: `daily/YYYY-MM-DD.md` with a starter template.
6. Print instructions: "Run `brain start` to begin."

### `brain start` Flow

1. Read `system/brain.config.yaml`.
2. Validate the configured agent CLI is installed and accessible (run `claude --version`, `codex --version`, or `gemini --version`).
3. Start Express server on configured port.
4. Serve `web/index.html` at `/`.
5. Handle WebSocket connections at `/ws`.
6. Log: "Brain is running at http://localhost:3000 (agent: claude-code)".
7. Open browser automatically (use `open` on macOS, `xdg-open` on Linux).

### V2 Setup Script (`deploy/setup.sh`)

The script should:

1. Install Node.js 22 via NodeSource.
2. Install `obsidian-headless` globally.
3. Install Claude Code globally (`npm install -g @anthropic-ai/claude-code`).
4. Prompt for Obsidian credentials → run `ob login`.
5. Prompt for vault name → run `ob sync-setup`.
6. Prompt for `CLAUDE_CODE_OAUTH_TOKEN` → write to `.env`.
7. Prompt for a web UI password → write to `.env`.
8. Clone the Brain repo, `npm install`.
9. Start `ob sync --continuous` and the Brain server via PM2.
10. Set up Cloudflare Tunnel (interactive) or print Tailscale instructions.
11. Print the access URL.

### V2 Auth

- Simple password-based middleware.
- On first visit, the user enters a password (set during deploy).
- Password stored as a bcrypt hash in `.env`.
- Session cookie set on successful login (httpOnly, secure).
- No user accounts, no OAuth — just one password for one user.

## Agent Authentication Reference

### Claude Code

```bash
# On your local machine (has a browser):
claude setup-token
# Completes OAuth flow, prints a 1-year token: sk-ant-oat01-xxxxx

# On the server (or in .env):
export CLAUDE_CODE_OAUTH_TOKEN="sk-ant-oat01-xxxxx"

# Verify:
claude -p "say hello" --output-format stream-json
```

- Token is valid for 1 year. Regenerate annually.
- Works with Pro, Max, Team, or Enterprise subscriptions.
- Does NOT require an API key. Uses your subscription.
- `claude -p` (the official CLI binary) on a VPS is permitted under Anthropic's terms for personal use.

### ChatGPT Codex

```bash
# Install:
npm install -g @openai/codex

# Auth (interactive, opens browser):
codex auth

# Or set API key:
export OPENAI_API_KEY="sk-..."

# Verify:
codex "say hello"
```

- Codex uses OpenAI API credits or ChatGPT Plus/Pro subscription.
- Check OpenAI's current docs for the latest auth flow.

### Gemini CLI

```bash
# Install:
npm install -g @anthropic-ai/gemini-cli
# (or whatever the current package name is)

# Auth:
gemini auth

# Verify:
gemini "say hello"
```

- Gemini CLI has a free tier with Gemini 2.5 Pro.
- Auth is Google account-based.

## Non-Goals (Explicitly Out of Scope)

- **No Obsidian plugin.** Brain is a standalone tool. Obsidian is just the vault.
- **No RAG / vector search.** The agent uses grep/glob to search the vault. Good enough for personal scale (~100 files).
- **No multi-model routing within a session.** You pick one agent at startup. Switching requires restart.
- **No calendar/email integration.** Brain manages knowledge, not scheduling.
- **No real-time collaboration.** Single user only.
- **No mobile app.** The web UI accessed from a mobile browser is the mobile experience.

## Open Questions (For the Developer)

1. **Streaming format differences:** Claude Code's `--output-format stream-json` emits structured JSON chunks. Codex and Gemini CLI may stream differently (raw text, SSE, etc.). The agent spawner needs to normalize these into a common format for the WebSocket.

2. **Long-running agent operations:** If the agent decides to read 10 files, edit 3, and run a grep — this can take 30-60 seconds. The UI should show a "thinking..." state and stream partial output as it arrives.

3. **Conversation history size:** Injecting the full conversation history into every prompt will eventually hit context limits. Implement a sliding window (last 10 turns) + compact summary of earlier turns.

4. **Concurrent sessions:** V1 can assume single session (one browser tab). The server should reject a second WebSocket connection or queue it.

5. **System prompt size:** `CLAUDE.md` could grow large as the user customizes it. Monitor token usage and warn if the system prompt exceeds ~2000 tokens.
